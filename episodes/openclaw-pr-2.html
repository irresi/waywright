The Emoji That Counted as 1.4 Characters

A debug log calls `.length` honest character counting, but it's counting UTF-16 code units, and emoji don't play by that rule.

· merged 2026-07-17
cold open

A debug log at src/agents/cli-runner/execute.ts:158 prints promptChars=${params.prompt.length} every time a CLI run kicks off. Someone pastes a prompt containing hi 🚀🎯 and the log says promptChars=7. Count the visible characters yourself: h, i, space, 🚀, 🎯 — that's 5. The mismatch is UTF-16 code units vs codepointsUTF-16 code units vs codepoints — `.length` counts UTF-16 code units, not characters; astral codepoints (most emoji) take 2 units.: .length counts code units, and each emoji here is a surrogate pair worth two units.

incident

This isn't a one-off typo. Issue #1 flagged the identical pattern in src/agents/subagent-announce.ts, where a debug log labels findings.length and triggerMessage.length as findingsChars/triggerMessageChars. Both logs make the same claim: this number is a character count. Both are wrong whenever the text has emoji.

investigation

irresi checked whether issue #1's exact file was reachable on main and found it wasn't — that announce body log lives on an unmerged upstream branch. But the same code-unit-labeled-as-Chars pattern exists on main right now, at execute.ts:158. Fixing a file that isn't merged yet fixes nothing; fixing the live instance fixes something real.

prediction
🔮 your call — before the reveal

Will switching promptChars from .length to [...prompt].length change the logged count for '안녕하세요'?

⬩ the reveal unlocks when you commit to a diagnosis ⬩
reveal

One line changes: promptChars=${params.prompt.length} becomes promptChars=${[...params.prompt].length}. Spreading the string iterates codepoints instead of UTF-16 code units. For 'hi 🚀🎯', that's 7 → 5. For 'loop 🔁 완료', 10 → 9. For pure BMP text like '안녕하세요', 5 → 5 — no change, no regression.

code walk

The whole fix, in context.

src/agents/cli-runner/execute.ts
-        `cli exec: provider=${params.provider} model=${context.normalizedModel} promptChars=${params.prompt.length}`,
+        `cli exec: provider=${params.provider} model=${context.normalizedModel} promptChars=${[...params.prompt].length}`,

Spread the string before taking .length so surrogate pairs collapse back to one count each.

resolution

Debug-log only, no behavioral change for users. The count promptChars reports now matches what a human counting characters by eye would get, for any input.
